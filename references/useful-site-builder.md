# Useful Site Builder

Use this reference during phase 4 to build the primary website experience. Choose the site pattern that best fits the niche instead of defaulting to a mini tool.

## Site Patterns

| Pattern | Best For | Must Include |
| --- | --- | --- |
| Tool site | calculators, converters, generators, checkers | working interaction, examples, assumptions, edge cases |
| Template library | resumes, emails, Notion, docs, prompts | downloadable/copyable templates, categories, usage guidance |
| Directory site | software, resources, datasets, services | curation criteria, filters, item pages, comparison fields |
| Comparison hub | SaaS alternatives, product choices, methods | fair criteria, comparison tables, recommendation logic |
| Checklist hub | visa, moving, school, business setup | step lists, timelines, printable or copyable formats |
| Tutorial hub | coding errors, Excel, software workflows | step-by-step instructions, examples, troubleshooting |
| Hybrid site | niches with multiple intents | one primary utility plus supporting content clusters |

## Build Requirements

Every site must include:

- A clear homepage promise
- One primary useful experience, not only articles
- At least 10 supporting pages with distinct intent
- Internal links between tool/template/directory pages and guides
- Mobile-friendly layout
- Accessible controls and readable typography
- Trust pages from `eeat-site-builder.md`
- Technical SEO basics from the readiness audit

## Pattern-Specific Requirements

For tool sites:

- Include clear input controls, result output, examples, and assumptions.
- Test at least three input cases.

For template libraries:

- Make templates copyable or downloadable.
- Include preview, use case, customization notes, and common mistakes.

For directories:

- Define inclusion criteria.
- Include filters or sortable fields when useful.
- Avoid fake listings, scraped descriptions, or unverifiable claims.

For comparison hubs:

- Explain evaluation criteria.
- Include pros, cons, best-for cases, and decision rules.
- Avoid pretending to have tested products unless evidence is provided.

For checklist hubs:

- Organize by phase, deadline, or user situation.
- Provide concise action items and caveats.

For tutorial hubs:

- Include prerequisites, steps, expected result, common errors, and related guides.

## Completion Gate

Before marking the build complete:

- Run the app or static preview if possible.
- Check that the primary site experience works without ads.
- Confirm no page exists only to target a keyword.
- Confirm the site has a reason to be bookmarked, shared, or revisited.
