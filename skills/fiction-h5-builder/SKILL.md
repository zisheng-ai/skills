# Fiction H5 Builder

## Overview

Create reader-facing novel H5 websites with strong mobile reading UX, polished editorial aesthetics, internationalization, and desktop adaptation. Treat writing tools such as oh-story as upstream content sources only; the generated site should feel like a real reading product, not an AI workflow demo.

## Workflow

1. Confirm the output surface: static H5 preview, Next.js template, existing repo change, or Sites artifact.
2. If the user provides a repo, inspect the existing stack and preserve its conventions.
3. If building from scratch, prefer the simplest viable implementation:
   - Static HTML/CSS/JS for a lightweight H5 prototype.
   - Next.js + Tailwind/vanilla CSS when routing, data loading, SEO, or future CMS integration matters.
4. Load only the references needed for the task:
   - Design planning and visual identity: `references/design-system.md`.
   - Product surface and IA: `references/product-surface.md`.
   - Reading UX: `references/reader-ux.md`.
   - Visual quality: `references/mobile-ui.md`.
   - Internationalization: `references/internationalization.md`.
   - oh-story/Markdown data shape: `references/data-contract.md`.
   - QA and screenshots: `references/qa-checklist.md`.
5. When the site must load novels generated by `worldwonderer/oh-story-claudecode`, treat `references/data-contract.md` as required and either implement an equivalent loader or adapt `scripts/parse-oh-story.mjs`.
6. Before coding any new UI, create an internal design plan using `references/design-system.md`: purpose, audience, tone, palette, type system, layout concept, and one signature element.
7. Build the usable product first. Do not create a marketing landing page unless requested.
8. Verify mobile first, then desktop. Fix visual density, clipping, hierarchy, and reader comfort before finalizing.

## Non-Negotiables

- Reader-facing only by default: no AI labels, no writing workflow, no creator backend, no "generated by" messaging.
- H5/mobile experience takes priority over desktop, but desktop must be intentionally adapted.
- Required pages for a complete template: home/work list, book detail with chapter catalog, reader.
- Do not add ranking, category, bookshelf, history, search, payment, comments, or account modules unless explicitly requested.
- Required reader controls: font size, line height or density, theme/night mode, chapter catalog, progress, previous/next chapter.
- Respect the content language: set `lang`, choose language-aware font stacks, and avoid fixed text widths that break long English/Spanish words or CJK line flow.
- Avoid cheap web-novel visuals: loud gradients, fake glass panels, excessive shadows, tiny cramped text, ad-card clutter, low-contrast text, and novelty fonts for body copy.
- Make one deliberate, brief-specific visual choice. Everything else should support reading comfort.

## Collaboration With Other Skills

If a frontend design, Sites, static app preview, or framework-specific skill is available, use it for implementation and visual verification. Still enforce this skill's product, reading, language, and QA rules as the source of truth for novel reading sites.

When a frontend-design skill is available, use it as an additional critique pass for aesthetic direction and visual distinctiveness. Do not let it override the reader-first constraints in this skill.

## Output Standards

- Use realistic book data and chapter excerpts, not placeholder lorem ipsum.
- If oh-story content is present, load real `正文/*.md` or `短篇/*/正文.md` files before falling back to mock data.
- Include empty/loading/error states when building an app template.
- Use semantic HTML for reading content: `article`, `h1`, `nav`, `main`, ordered chapter lists, and accessible buttons.
- Keep reader preferences local only for prototypes. For real products, model reading progress as durable data only when requested.
- For desktop, avoid simply stretching H5. Use a centered reader column, optional side catalog, and restrained max widths.

## Verification

Before final response, run the relevant build/typecheck/lint commands if available. For visual work, use browser screenshots at minimum:

- 390 x 844 mobile home.
- 390 x 844 mobile reader.
- 430 x 932 mobile reader.
- 1366 x 900 desktop home.
- 1366 x 900 desktop reader.

If browser verification is unavailable, state that clearly and still perform static checks.
